## 브랜치 전략

### 브랜치 명명 규칙

```
feature/기능-이름          # 새로운 기능 개발
bugfix/버그-이름           # 버그 수정
refactor/리팩토링-이름     # 코드 리팩토링
docs/문서-이름             # 문서 작성/수정
chore/작업-이름            # 빌드, 의존성 등 관리
```

**예시:**
- `feature/user-authentication`
- `bugfix/login-error-handling`
- `refactor/api-layer`
- `docs/setup-guide`

### 브랜치 생성 규칙

1. **반드시 최신 main 브랜치에서 시작**
   ```bash
   git checkout main
   git pull origin main
   git checkout -b feature/your-feature-name
   ```

2. **각 기능마다 별도의 브랜치 생성**
   - 여러 기능을 하나의 브랜치에서 개발하면 안 됩니다
   - 한 브랜치는 하나의 기능 또는 버그 수정만 담당합니다

## 커밋 컨벤션

### 커밋 메시지 형식

```
<타입>(<범위>): <제목>

<본문>

<푸터>
```

### 타입 정의

| 타입 | 설명 | 예시 |
|------|------|------|
| `feat` | 새로운 기능 추가 | `feat(auth): 로그인 기능 구현` |
| `fix` | 버그 수정 | `fix(api): 사용자 조회 에러 수정` |
| `docs` | 문서 작성/수정 | `docs: README 업데이트` |
| `style` | 코드 포맷팅, 세미콜론 등 | `style: 코드 포맷팅 정리` |
| `refactor` | 기능 변화 없는 코드 리팩토링 | `refactor(database): ORM 마이그레이션` |
| `test` | 테스트 추가/수정 | `test(api): 사용자 조회 테스트 추가` |
| `chore` | 빌드, 의존성, CI/CD 등 | `chore: 패키지 업데이트` |
| `perf` | 성능 개선 | `perf(cache): 쿼리 캐싱 추가` |

### 커밋 메시지 작성 가이드

**제목 (Subject)**
- 50자 이내로 작성
- 명령형으로 작성 ("수정했습니다" X → "수정" O)
- 첫 글자는 대문자
- 마침표로 끝내지 않기

**본문 (Body)**
- 72자 마다 줄바꿈
- "무엇을"과 "왜"를 설명
- 개행으로 제목과 본문 구분

**푸터 (Footer)**
- 이슈 트래킹 또는 breaking changes 기록
- `Closes #123` (이슈 종료)
- `BREAKING CHANGE: 설명` (하위 호환성 깨질 때)

### 커밋 메시지 예시

```
feat(auth): JWT 토큰 검증 기능 추가

사용자 로그인 시 발급한 JWT 토큰을 검증하는
미들웨어를 추가했습니다.

- 토큰 만료 시간 체크
- 토큰 서명 검증
- 유효하지 않은 토큰 처리

Closes #45
```

## PR 프로세스

### Step 1: 브랜치에서 개발 진행

```bash
# 브랜치에서 기능 개발
git add .
git commit -m "feat(feature-name): 설명"
```

**중요: 진행 상황을 자주 커밋하세요**
- 작은 단위의 커밋으로 진행 상황을 공유
- 비상시 롤백이 용이하도록 리비전 생성
- 최소 2개 이상의 의미있는 커밋 필요

### Step 2: 원격 저장소에 푸시

```bash
git push origin feature/your-feature-name
```

### Step 3: PR 생성

1. GitHub에서 PR 생성
2. **PR 템플릿을 반드시 따르기**
3. 제목은 명확하고 간결하게
4. 체크리스트를 모두 확인

### Step 4: 코드 리뷰

- PR이 생성되면 팀원들의 리뷰를 기다립니다
- 리뷰 피드백에 응답하고 수정합니다
- 모든 회담이 해결될 때까지 머지하지 않습니다

### Step 5: 머지

1. 모든 리뷰 승인
2. CI/CD 통과
3. "Squash and merge" 또는 "Create a merge commit" 선택
4. 브랜치 삭제

## 코드 리뷰

### 리뷰어의 책임

- 코드 품질 검증
- 컨벤션 준수 확인
- 버그 및 보안 이슈 체크
- 개선 사항 제안

### 리뷰 시 확인 사항

- [ ] 코드가 컨벤션을 따르는가?
- [ ] 테스트 코드가 충분한가?
- [ ] 문서화가 되어있는가?
- [ ] 성능 이슈는 없는가?
- [ ] 보안 이슈는 없는가?

### 피드백 예절

- 존중하는 톤으로 피드백
- "왜 이런 결정을 했나요?" 형태로 질문
- 개선 방안을 제시하기
- 칭찬도 아끼지 않기

## 롤백 가이드

실수로 병합된 코드를 롤백해야 할 경우:

```bash
# 이전 커밋으로 돌아가기
git revert <커밋-해시>

# 또는 특정 PR 전체를 되돌리기
git revert -m 1 <머지-커밋-해시>
```
